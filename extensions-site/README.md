# extensions-site

A DuckDB-driven landing site for the **ducklink** extension registry, served by
`ducklink serve`. Lists the catalog (component extensions + DuckDB builtins +
official out-of-tree extensions) with a client-side search bar and a
per-extension detail page; also exposes the catalog as a JSON API and serves the
built `.wasm` component artifacts directly from the database (in-DB CAS).

```
   /                          landing page with search
   /ext/<name>                per-extension detail
   /api/extensions.json       full registry as JSON
   /api/ext/<name>.json       per-extension as JSON
   /asset/<name>              raw component .wasm bytes (content-addressed)
   /asset-info/<name>         { name, size_bytes, sha256, content_type, url }
   /static/ducklink_logo.png  the logo
   /health                    ok
```

Everything is served from a single **DuckDB** file (`registry.db`) generated by
`build.py` from `registry/index.json` (the canonical catalog) and the built
artifacts under `artifacts/extensions/`. Route handlers live in `registry.db`'s
`routes` table ŌĆö the same table `ducklink serve`'s httpd consults on every
request. Each handler is DuckDB SQL (or a static body) that materializes HTML or
JSON; pages + JSON are baked at build time, so a fully-lean core (just
`core_functions` + `parquet`) serves the whole site.

## Local development

```bash
# 1. Build the component artifacts you want listed/downloadable
make sample-extension        # or: cargo component build -p isin-component ...
# (artifacts land in artifacts/extensions/*.wasm)

# 2. Generate the site DB (requires the duckdb python package pinned to the
#    core's storage version so `ducklink serve` can open the file)
python3 -m pip install 'duckdb==1.4.0'
python3 extensions-site/build.py
# -> wrote extensions-site/registry.db ŌĆö 31 extensions, 3 artifacts, ... bytes

# 3. Serve locally on http://localhost:8080
make host                    # builds ./target/release/ducklink + the core component
./target/release/ducklink serve --db extensions-site/registry.db --port 8080
```

The browse + search experience embeds the catalog JSON in the landing page (no
XHR), so it works offline. The `/asset/<name>` route streams the component
`.wasm` byte-exact out of the `assets` table (verified: the SHA-256 of
`curl /asset/isin` matches `assets.sha256`).

> Storage-version note: the wasm core is DuckDB 1.4.0, so `build.py` must write a
> 1.4.0-format file ŌĆö install `duckdb==1.4.0` (a newer writer produces a file the
> 1.4.0 core can't open).

## Architecture: why a DuckDB database is the whole site

`ducklink serve` (see `docs/duckdb-wasm-httpd.md`) consults the `routes` table
on every request, binding `$method` / `$path` / `$query` / `$body` / `$remote`.
The result row's `body` / `status` / `ctype` columns become the HTTP response.

- The landing page is one `static` route whose body is the full HTML (data baked
  in at build time).
- The per-extension page is a `sql` route: `SELECT html FROM pages WHERE
  name = regexp_replace($path, '^/ext/', '')`, with a `CASE` for the 404.
- The JSON APIs are pre-rendered (`static` landing JSON + a `pages`-style table
  for per-extension JSON) ŌĆö no `json`-extension functions needed at request time.
- The `.wasm` artifacts live in the `assets` table and are served by the `blob`
  route kind (`SELECT blob FROM assets WHERE name = ...`), streamed raw with
  `Content-Type: application/wasm`.

No external object store, no egress fees, no DNS to coordinate ŌĆö one DuckDB file
is the catalog, the website, and the artifact CDN.

### Schema (the distribution database)

```sql
CREATE TABLE routes        (method, pattern, handler, kind, status, ctype, priority);
CREATE TABLE pages         (name VARCHAR PRIMARY KEY, html VARCHAR);   -- per-ext HTML
CREATE TABLE api_ext       (name VARCHAR PRIMARY KEY, json VARCHAR);   -- per-ext JSON
CREATE TABLE extensions    (name, kind, status, description, version, repository, ...);
CREATE TABLE assets        (name VARCHAR PRIMARY KEY, blob BLOB, size_bytes, sha256,
                            content_type, info_json);                  -- in-DB CAS
CREATE TABLE static_assets (name VARCHAR PRIMARY KEY, blob BLOB, content_type);  -- logo
CREATE TABLE site_meta     (key VARCHAR PRIMARY KEY, value VARCHAR);
```

## Container build

Unlike sqlite-wasm's site (a self-contained binary), `ducklink serve` runs the
**wasm core component**, so the runtime image needs three prebuilt artifacts: the
native `ducklink` binary, `ducklink_core.wasm`, and `registry.db`. Build
them on the host first (`make host` + `python3 extensions-site/build.py`), then:

```bash
docker build -t ducklink-extensions-site -f extensions-site/Dockerfile .
docker run --rm -p 8080:8080 -e PORT=8080 ducklink-extensions-site
```

The `ducklink` binary is compiled inside the image (a normal Rust binary linking
wasmtime ŌĆö no wasm toolchain needed); the core `.wasm` is copied from the host
tree (it requires the wasi-sdk + libduckdb pipeline, which isn't Docker-friendly,
so it must be prebuilt). `cloudbuild.yaml` deploys the image to Cloud Run.

### Updating the catalog

Edit `registry/index.json` (or build new extensions), re-run
`python3 extensions-site/build.py`, and re-serve / re-deploy. `registry.db` is
generated and git-ignored.
